Visualizing Boston Flood Zones Due to Sea Level Rise: GIS Technology for Map Geospatial Data¶

Sea level rise is an increasing problem facing many coastal communities in the US and around the world. According to the following article, "by 2100, high tide flooding will occur at least every other day in the USA."

http://floodlist.com/america/usa/noaa-report-high-tide-flooding-increase

Boston is not exempt from the threat of flooding due to sea level rise. The MassDOT-FHWA project predicts a sea level rise of 9 inches by the middle of this century, 21 inches in the second half of this century, and 36 inches towards the end of this century.

How will sea level rise affect flooding in Boston? In this lesson, we will use Geographic Information System (GIS) mapping technology to map Boston flood zones for varying sea level rise scenarios projected over the rest of the century, and use the maps to answer this question.

Data¶

The data are available from Analyze Boston, the City of Boston open data hub. The flood zone data for various sea level rise scenarios are provided in the following links.

  • 9INCH SEA LEVEL RISE 1PCT ANNUAL FLOOD
  • 9INCH SEA LEVEL RISE 10PCT ANNUAL FLOOD
  • 21NCH SEA LEVEL RISE 1PCT ANNUAL FLOOD
  • 21INCH SEA LEVEL RISE 10PCT ANNUAL FLOOD
  • 36NCH SEA LEVEL RISE 1PCT ANNUAL FLOOD
  • 36INCH SEA LEVEL RISE 10PCT ANNUAL FLOOD

The risk of flooding in coastal and riverine regions of Boston was created from computer simulations of complex weather models, coupled with flood hazard models based on 9, 21 and 36 inches of sea level rise. For each amount of sea level rise, the hazard model also tells us the zones that have a 1% (1-in-100) and 10% (1-in-10) chance of flooding in any given year. A full description of each data set is provided in each link.

The Shapefile Format¶

The format of the flood zone data is provided as shapefiles. Shapefiles are a way of encoding geospatial information, developed by the Environmental Systems Research Institute, Inc. (ESRI). In fact, a shapefile consists of three files: a main file (.shp), an index file (.shx), and a dBASE table (.dbf). You can check to see these three files with the same name and respective file extensions in the folder where you unzipped the downloaded .zip file into.

In our flood zones example, the shapefile encodes the flood zones as polygons, and it provides the latitude and longitude of the polygon corners. Recall that the latitude of a point on earth indicates how far north or south of the equator it is, and the longitude indicates how much east or west of the prime meridian it is. For reference, Boston is approximately 42.36$^\circ$N, 71.06$^\circ$W. We'll write it as a (Longitude, Latitude) pair, $(-71.06, 42.36)$, where a negative longitude means west of the prime meridian.

In general, shapefiles can contain a lot of other types of geographic information, such as

  • Multiple shapes of various types (points, lines, polygon, etc.). These data are stored in the main .shp file.
  • Feature attributes of each shape. For example, if a shape represents a county, the feature attributes might contain the county name, population, etc. These data are stored in the dBASE .dbf file.

Further reading: Gentle introductions to shapefiles can be found on many blogs and online tutorials, such as this. Full technical details of the ESRI shapefile can be found in their white paper: ESRI Shapefile Technical Description.

There is only 1 shape recorded in our shapefile, which in this case is the flood zone itself. In general, a shapefile can contain multiple shapes, such as a shapefile of the US with each State being one shape. What comprises a shape can be fairly arbitrary and dependent on the nature of the geospatial data.

Each shape can comprise of multiple parts. A simple mental picture to understand why there can be many parts is to imagine many lakes in a State---one part is a polygon outlining the State border and the other parts are polygons outlining the interior lakes. Note that the parts are also allowed to be of different shape type.

Returning to our flood zone example, each part in the shape is a polygon. We'll define helper functions that will return the number of parts in a shape and retrieve a part. If you're interested to know how the helper functions work, head to the appendix for details.

In [9]:
# A helper function to return the number of parts in the shape
def get_num_parts(shape):
    return len(shape.parts) - 1

# A helper function to retrieve a polygon part from the shape.
def get_part(shape, part):
    polygon = shape.points[ shape.parts[part] : shape.parts[part+1] ]
    return polygon

Overlaying the Flood Zone on Google Maps¶

The mapping of the flood zone can be made more user-friendly if it is overlaid on an easily recognizable map. Let's do this using Google Maps---it is fast and easy with the Google Maps API and the Python module gmplot, and also interactive!

Overlaying the flood zone on Google maps is easy: first, we retrieve the map of Boston. Then, we plot the polygons directly onto the map. Finally, we draw the map into an HTML file for interactive viewing. The following code implements these steps.

In [17]:
from gmplot import gmplot

# (1) Get a map of Boston
Boston_map = gmplot.GoogleMapPlotter.from_geocode("Boston") # Google Maps API knows where Boston is!
# (Note: gmplot sometimes complains about the above line of code. If that happens, just rerun it.)


# The following 3 lines addresses a technical point of using a Google Maps API authentication key (see below)
fcon = open("./~APIkey_localhost_8888", "r")
Boston_map.apikey = fcon.readline()
fcon.close()


# (2) Plot each polygon part on the map
for part in range(nparts):
    lons, lats = zip(*get_part(shape, part))
    Boston_map.plot(lats, lons, 'b')        # Plot the polygon directly onto the map. Note (lat, lon) not (lon, lat)!


# (3) Draw the map into an HTML file
Boston_map.draw("Boston_Map_With_Flood_Zone.html")

A technical point about authenticating the Google Maps API: Google requires an authentication key to allow the Javascript applications in the HTML file to access Maps API. In order to make this code work from Jupyter Notebook, I've created an API key that works only for localhost:8888 servers, and saved it to a local file named ~APIkey_localhost_8888. These three lines of code above opens the file, reads the key into the Boston_map object, and then closes the file. If you are not using Jupyter Notebook, remove these three lines of code so that you can open the HTML file directly from your local computer.

Appendix¶

The get_num_parts and get_part functions¶

Earlier, we had defined the helper functions get_num_parts and get_part. Here's an explanation of the shapefile data structure and how those functions work.

Recall that shape = flood_zone.shape(0) is the (sole) record in the shapefile, and has all the information about each part of the shape. The two important variables in shape are

  • shape.parts
  • shape.points

The latter, shape.points, is simply a list of (Longitude, Latitude) pairs. The first 10 points are shown below.

Then, shape.parts encodes a way to tell us which points belong to which part. The first 4 entries are shown below.

In [23]:
shape.parts[:4]
Out[23]:
array('i', [0, 5, 10, 15])
  • Part 0 contains the points from 0 (inclusive) up to 5 (exclusive), i.e., shape.points[0:5]. 0 is shape.parts[0] and 5 is shape.parts[1].
  • Part 1 contains the points from 5 (inclusive) up to 10 (exclusive), i.e., shape.points[5:10]. 5 is shape.parts[1] and 10 is shape.parts[2].
  • Part 2 contains the points from 10 (inclusive) up to 15 (exclusive), i.e., shape.points[10:15]. 10 is shape.parts[2] and 15 is shape.parts[3].

In general,

Part n contains the points from shape.parts[n] (inclusive) up to shape.parts[n+1] (exclusive), i.e.,
shape.points[shape.parts[n] : shape.parts[n+1]].

Finally, the total number of parts must be one less than the length of shape.parts (otherwise shape.parts[nparts+1] will not make sense).
